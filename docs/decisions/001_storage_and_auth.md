# 001_データの保存場所と権限制御（認証）の必要性についての検討

*   **日付**: 2026-04-16
*   **ステータス**: 決定済（方針確定）
*   **関連課題**: バックログ #10（権限管理の追加）

## 1. 背景と課題 (Context)
現在の「ダイエットダッシュボード (v1.0.0)」では、体重などのユーザー記録をブラウザの標準機能である `LocalStorage` を用いて保存している。

> [!NOTE]
> **用語解説: LocalStorage (ローカルストレージ)**
> Webブラウザ（ChromeやSafariなど）に備わっているデータの保存領域のこと。サーバー（インターネット上）ではなく、ユーザー自身が使っているスマホやPCの中に直接データが保存されます。

これに対し、以下の2点の仕様・課題が認識された。

1.  **データ消失・非同期の課題**: LocalStorageはブラウザごとに紐付いた保存領域であり、スマホとPCなど「複数端末での同期」ができず、またブラウザのキャッシュクリア等でデータが消失するリスクがある。
2.  **第三者によるデータ書き換えリスク**: Web上に公開されているため、今後クラウドベースでデータ同期を行うようにした場合、誰でもデータ（体重等）を書き換えられてしまう可能性がある。

## 2. 検討した選択肢 (Options)

### 選択肢A: 現状維持（LocalStorageのままパスワードロック）
*   **メリット**: 実装が容易。バックエンド層（データベース等）との連携が不要。
*   **デメリット**: 根本的な課題である「複数端末間でのデータ同期」や「データ消失リスクの軽減」が解決しない。

### 選択肢B: クラウドDB＋認証機能の導入（Firebase等） [★採用方針]

> [!NOTE]
> **用語解説: クラウドDB / Firebase / 認証機能**
> *   **クラウドDB (Database)**: インターネット上にデータを預ける仕組み。ここに保存すれば、スマホからでもPCからでも同じデータを取り出して見ることができます。
> *   **Firebase (ファイアベース)**: Googleが提供している、アプリ開発を急ピッチで進めるための便利なツール群（DBや認証などを簡単に組み込める）。
> *   **認証 (Authentication)**: 「あなたは誰ですか？」を確認する機能（ログイン）。データベースと組み合わせることで、「このデータはこの人しか見れない・書けない」といった鍵（アクセス制御）をかけられます。

データを端末ローカルではなく、インターネット上のデータベース（Firebase Firestoreなど）に保存し、同時に「ユーザー認証（ログイン）」機能を実装する。
*   **メリット**:
    1. どの端末（PC、スマホ）からアクセスしても同じデータを閲覧・編集できる。
    2. ログインした自分だけしかデータを書き換えられない権限（アクセス制御）を担保できる。
    3. 端末の破損やブラウザの初期化によるデータロストを防げる。
*   **デメリット**: バックエンドサービスであるFirebaseとの連携作業が必要になり、やや実装規模が大きくなる。

## 3. 決定事項とこれからの対応 (Decision & Action)

中長期的な運用の安定性と安全性を考慮し、**「選択肢B：クラウドDB（Firebase）および認証機能の導入」**を将来的なアーキテクチャ方針として採用する。

**直近のアクション:**
*   バックログ #10 として「権限管理（編集制限）機能の追加」を優先度：高 で登録済。
*   まずは現在のLocalStorageアーキテクチャのまま、簡単な「フロントエンドでの編集保護（パスワード等）」を一次対応として挟むか、一気にFirebase連携（クラウド化＋Googleログイン等のAuth）を進めるかを次期作業時に判断する。
