# 開発ガイドライン (Contributing Guidelines)

本プロジェクト（ダイエットダッシュボード）を美しく健康的に開発・保守していくための「自分への約束事（ルール）」です。

## 1. アイデアの管理（バックログの活用）
* 思いついた要望、機能追加、不具合の修正アイデアは、**いきなりコードを書き始めない**こと。
* まずは `project_management/課題_バックログ管理表.md` の「進行中・オープンな課題 (Backlog)」にラフなメモとして追記する。
* 課題ID（例: `#03`）を振り、やるべきことが明確になってから着手する。

## 2. コミットメッセージの記法
後から歴史（Gitの履歴）を振り返ったときに「この時何をしたのか」が一目でわかるように、以下のプレフィックス（接頭辞）を必ずつけること。

* `feat:` （Feature）新しい機能の追加
* `fix:` バグや不具合の修正
* `docs:` システム設計書やREADMEなどドキュメントのみの更新
* `style:` CSSの変更、デザイン修正、コードのフォーマット変更
* `refactor:` リファクタリング（機能そのものは変えずに、内部のコードを整理した時）

**コミットメッセージの例**:
> `feat: スマホ向けに体重追加フォームを画面上部に移動`

## 3. ブランチ（枝分かれ）の運用ルール
* メインの開発・運用は常に `main` ブランチで行う。
* **大規模な新機能**（例: PWA化、大幅なデザイン刷新）など、途中で壊れるリスクがある場合は、一時的に新しい作業用ブランチ（例: `feature/design-refresh`）を切ってから開発を行う。
* 区切りが良い機能の完成時には、`main` に合流させたのち、必要に応じてタグやアーカイブブランチを作成し、常に「動く状態の過去」を振り返れるようにする。
